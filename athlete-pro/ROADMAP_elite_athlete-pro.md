# Road to Elite — Athlete Pro

## Этап 1. Фундамент (контрольная точка 0)

**Цель**: всё стабильно работает, легко запустить, легко проверять.

- Убедиться, что на любой машине можно:
  - клонировать репозиторий;
  - выполнить `npm ci`;
  - запустить `npm test`, `npm run lint`, `npm run format:check`;
  - запустить `npm start` или `npm run start:3100` и открыть приложение.
- В `README.md` зафиксировать мини-гайд для себя:
  - “Как я проверяю, что ничего не сломал перед пушем”.

## Этап 2. Архитектура фронтенда

**Цель**: код можно развивать годами, не боясь трогать старые файлы.

- Разбить фронт на зоны:
  - `Dashboard`, `Workout`, `Profile`, `Analytics`, `BodyStats`, `AI Coach`.
- Постепенно разбивать большие файлы (`dashboard.js`, `workout.js`, `profile.js`, `claude.js` и др.) на модули:
  - чистые функции (подсчёт тоннажа, формулы, проверки);
  - константы (цвета, словари мышц, справочники упражнений).
- Ввести мини-паттерн `state + view` без фреймворка:
  - отдельный слой состояния (данные + операции над ними);
  - отдельный слой отрисовки (функции, которые обновляют DOM).
- Использовать ИИ как помощника:
  - просить разбиения файла на state/view/helpers;
  - просить выносить чистые функции и писать к ним тесты;
  - всегда разбирать, что именно он предлагает и зачем.

## Этап 3. Архитектура бэкенда

**Цель**: Express‑сервер готов к росту (логирование, авторизация, новые API).

- Разобрать `server.js` на слои:
  - `routes/apiCoach.js`, `routes/status.js`, `routes/firebase.js` и т.п.;
  - `lib/anthropicClient.js` для работы с Anthropic;
  - в `server.js` оставить только создание `app`, подключения middleware и запуск.
- Для каждого эндпоинта описать контракт:
  - ожидаемый запрос;
  - формат ответа;
  - возможные ошибки (коды + JSON).
- С помощью ИИ:
  - сгенерировать аккуратную структуру `/routes` и `/lib` без смены поведения;
  - добавить JSDoc/описания типов для payload’ов.

## Этап 4. Дизайн‑система и визуальный уровень

**Цель**: приложение выглядит “дорого” и единообразно.

- Ввести дизайн‑токены:
  - CSS custom properties для цветов, шрифтов, размеров;
  - базовую сетку отступов (например, `4/8/16/24/32`).
- Вынести общие компоненты:
  - кнопки, карточки, поля ввода, модалки.
- Настроить типографику:
  - 2–3 уровня заголовков;
  - базовый текст и подписи;
  - единая иерархия на всех экранах.
- Адаптив:
  - пройтись по ключевым экранам (мобильный, планшет, десктоп);
  - решить, как ведут себя длинные списки, таблицы, графики.
- ИИ как арт‑директор:
  - попросить дизайн‑систему в одном CSS‑файле с нужным настроением (строгий зал, неоновые акценты);
  - попросить варианты анимаций для кнопок/карточек.

## Этап 5. Умный UX и микровзаимодействия

**Цель**: каждое действие ощущается продуманным и отзывчивым.

- Выделить 3–4 ключевых сценария:
  - создать тренировку;
  - посмотреть прогресс;
  - получить совет AI Coach;
  - обновить профиль/статы.
- Для каждого сценария:
  - убрать лишние шаги;
  - сделать понятные кнопки и тексты;
  - добавить явную обратную связь (лоадеры, toast’ы, подсказки).
- Добавить лёгкие микровзаимодействия:
  - анимации появления карточек;
  - плавные прогресс‑бары;
  - анимированные табы/переключения.
- Использовать ИИ:
  - просить идеи микро‑анимаций с учётом производительности;
  - улучшать пустые состояния (empty states).

## Этап 6. AI Coach уровня “про”

**Цель**: коуч даёт структурированные, персонализированные рекомендации.

- Укрепить system prompt (`_buildSystemPrompt`):
  - добавить учёт долгосрочных трендов, а не только окна 72 ч;
  - ввести режимы: восстановление, набор массы, сила, поддержание;
  - задать чёткий стиль ответов (структурированный, без воды).
- Расширить контекст:
  - история травм/ограничений;
  - предпочтения по упражнениям;
  - ограничения по времени и инвентарю.
- С помощью ИИ:
  - оптимизировать prompt под разные цели (силовые, бодибилдинг, здоровье);
  - спроектировать схему данных истории тренировок под хороший контекст для модели.

## Этап 7. Наблюдаемость и качество

**Цель**: понимать, что происходит, когда что‑то идёт не так.

- Логирование:
  - на backend: ошибки `/api/coach`, таймауты, неправильные payload’ы;
  - минимальный correlation ID на запрос;
  - на frontend: обработка fetch‑ошибок и понятные сообщения пользователю.
- Метрики:
  - Lighthouse‑отчёты (ручные или через CI);
  - простые “бизнес‑метрики” для себя: тренировки за неделю/месяц, прогресс по основным подъёмам.
- Использовать ИИ:
  - попросить список минимальных логов для Express‑приложения;
  - спроектировать простую схему event‑логов для пользовательских действий.

## Этап 8. Расширение фич и постоянное обучение

**Цель**: превратить проект в площадку для экспериментов и роста скиллов.

- Идеи фич:
  - расширенная аналитика (кластеры тренировок, отчёты, “узкие места”);
  - планировщик тренировочных циклов (4–6 недель с авто‑пересчётом нагрузок);
  - импорт/экспорт (JSON/CSV), синхронизация через Supabase/Firebase;
  - пользовательские шаблоны программ (сохранение/шаринг PPL‑схем);
  - онбординг для нового пользователя с 3–4 вопросами и стартовым планом.
- Для каждой новой фичи один и тот же цикл:
  1. Сформулировать цель.
  2. Продумать UX‑флоу.
  3. Спроектировать данные и API.
  4. Использовать ИИ как:
     - помощника в проектировании;
     - автора чернового кода;
     - напарника по ревью.
